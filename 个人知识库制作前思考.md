## 二. 关键技术深度解析

### 2.1 RAG (检索增强生成)
*   **定义**：在回答问题前，先从外部知识库检索相关信息，用于减少大模型的“幻觉问题”和“知识滞后”。
*   **核心配置项** (除了切片、索引，还有哪些？)：*   
    *   **分段大小 (Chunk Size)**：切片的粒度。
    *   **嵌入模型 (Embedding Model)**：文字 $\rightarrow$ 数字的过程。
    *   **召回数量 (Top K)**：每次检索返回多少条相关片段。
    *   **检索方式**：匹配策略。
    *   *主要在配置RAG时使用*
### 2.2 插件 (Plugins) 与 工作流 (Workflow)
*   **插件/工具**：联网搜索、绘图、代码、天气、股票等。
    *   *关键点*：联网搜索确实是由插件实现的。若无添加插件，则只能使用静态知识库或模型自带的训练数据。
*   **工作流**：
    *   **定义**：可视化拖拽，实现多步骤、条件判断、定义解释性业务或任务的自动化执行流程。
    *   **特点**：可视化编排，逻辑更灵活。

### 2.3 记忆机制 (Memory) 与 Redis 的必要性
*   **Bot 自带的记忆功能**：
    *   包含：用户偏好、对话历史，提供个性化服务。
*   **❓ 疑问**：当我使用 Bot 定制的智能体本身有“对话历史”的记忆功能，那为什么在二次开发中要用 Redis 进行历史记忆缓存？
*   **✅ 解答 (两个是不同层面的能力)**：
    1.  **作用范围不同**：Bot 自身的记忆功能只作用于当前对话/网页，**无法跨设备复用**。
    2.  **可控性差**：存储策略、过期时间、检索方式由平台决定，开发者不可控。
    3.  **数据隔离风险**：多用户下，平台记忆不好做到严格的数据隔离，容易出现**上下文串扰**。
    4.  **性能瓶颈**：平台自带记忆在大量并发下易成为性能瓶颈。
    *   **结论**：用 **Redis** 解决上述问题（持久化、隔离性、高性能）。

## 三. Bot 的搭建全流程 (SOP)

**平台**：Coze

### 第一阶段：创建与人设
1.  **创建**：填写名称、头像、描述。
2.  **设定人设与 Prompt (提示词/指令)**：
    *   **定义任务**：告诉 AI 它要干什么。
    *   **设定边界 (约束)**：回答范围、格式、风格。
    *   **提供上下文**：补充背景信息 (“基于上传文档”)。
    *   **引导逻辑**：结构化指令使 AI 按步骤完成复杂任务。

### 第二阶段：能力构建
3.  **选择大模型**。
4.  **搭建知识能力**：配置知识库、插件、工作流。
5.  **配置记忆与上下文**。

### 第三阶段：测试与发布
6.  **调试与预览**：测试可用性。
7.  **发布 (免费)**：
    *   可对其他人开源。
    *   支持发布为 API 和 网页。

### 第四阶段：后续开发 (搭建好 AI Agent 后)
*   **个人博客/学习心得/流程集成**。
*   **FastAPI 封装**：写接口，调用 Bot 接口。
*   **测试可用性**。
*   **最终发布**。
